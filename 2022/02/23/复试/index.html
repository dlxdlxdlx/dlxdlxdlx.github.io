操作系统

操作系统是什么?

操作系统是计算机系统中最基本的组成部分，它负责控制计算机的执行。
他也是计算机资源的管理者.

1.管理以下资源:

  • 处理机管理: (进程控制、进程同步、进程通信、死锁处理、处理机调度)
  • 处理器管理: (提高内存利用率，内存的分配与回收、地址映射、内存保护与共享、内存扩充)
  • 文件管理(计算机中的信息都是以文件的形式存在的)
  • 设备管理(完成用户的 I/O 请求，方便用户使用设备、并提高设备的利用率)

2.为用户提供使用计算机硬件系统的接口:

  • 命令接口 (用户通过控制台或终端输入操作命令，向系统提供各种服务要求)
  • 程序接口 (由系统调用组成，用户在程序中使用这些系统调用来请求操作系统为其提供服务)
  • 图形接口 最常见的图形用户界面 GUI（最终还是通过调用程序接口实现的)

3.为作为扩充机器

  • 将裸机改造成功能更强使用更方便的机器.

操作系统的特征

并发和异步是操作系统最基本的两个特征,互为存在条件.

  • 共享 系统资源可以由内存中的多个进程共同使用
  • 异步 进程的执行以一种不可知的速度推进
  • 虚拟 将一个物理上的实体变为若干个逻辑上的对应物
  • 并发 某一时间段内多个进程交替执行

CPU 运行机制

管态目态

CPU 执行两种不同性质的程序,一种是内核程序,另一种是用户自编程序.

  • 核心态: 核心态是用户态的管理者,可以执行特权指令.所谓特权指令，是指计算机中不允许用户直接使用的指令，如：I/O 指令、置中断指令，存取用于内存保护的寄存器，送程序状态字到程序状态字寄存器等指令。
  • 用户态: 操作受限,无法执行特权指令(除了陷入指令).
    • 用户态可以通过以下方式切换到核心态:
      • 系统调用.
      • 异常.
      • 外设中断.

内核

内核是计算机上配置的底层软件，是计算机功能的延伸，包括以下 4 个方面的内容：

  • 时钟管理
    时钟的第一功能是计时，操作系统需要通过时钟管理，向用户提供标准的系统时间。其次，通过时钟中断的管理，可以实现进程的切换。在分时操作系统中，采用时间片轮转调度的实现；在实时系统中，按截至时间控制运行的实现；在批处理系统中，通过时钟管理来衡量一个作业的运行程度等。
  • 中断机制 引入中断技术的初衷是提高多道程序运行环境中 CPU
    的利用率，主要针对外部设备。后来逐步得到发展，形成了多种类型，成为操作系统各项操作的基础。如，键盘或鼠标信息的输入、进程的管理和调度、系统功能的调用、设备驱动、文件访问等。都依赖于中断机制。可以说，现代操作系统是靠中断驱动的软件。中断机制中，只有一小部分功能属于内核，负责保护和恢复中断现场的信息，转移控制权到相关的处理程序。这样可以减少中断的处理时间，提高系统的并行处理能力。
  • 原语 操作系统底层是一些可被调用的公用小程序，它们各自完成一个规定的操作，其特点是：
    • 它们处于操作系统的最底层，是最接近硬件的部分.
    • 这些程序的运行具有原子性，其操作只能一气呵成.
    • 这些程序的运行时间都较短，而且调用频繁.
    • 定义原语的直接方法是关闭中断，让它的所有动作不可分割地进行完再打开中断。
  • 系统控制的数据结构及处理 系统中用来登记状态信息的数据结构很多，比如：作业控制块、进程控制块、设备控制块、各类链表等。
    • 为了实现有效的管理，系统需要一些基本的操作，常见的操作有以下三种：
    • 进程管理：进程状态管理、进程调度和分配、创建和撤销进程控制块等。
    • 存储器管理：存储器的空间分配和回收、内存信息保护程序、代码对换程序等。
    • 设备管理：缓冲区管理、设备分配和回收等。

中断和异常

中断: 来自 CPU 执行指令外的事件发生.如设备发出的 I/O 结束中断、时钟中断等。这一类中断通常是与当前执行的指令无关的事件。
异常: 源自 CPU 执行指令内部的事件.如程序的非法操作码、地址越界、算术溢出、缺页异常等。对异常的处理一般要依赖于当前程序的运行现场，不能被屏蔽。

中断的处理

  • 关中断: CPU 响应中断后,首先保护程序现场.屏蔽高优先级中断源请求.
  • 保存断点: 保证中断服务程序执行完成后能正确返回,需要保存程序计数器(PC)
  • 中断服务程序寻址: 取出中断服务程序入口地址送入 PC
  • 保存现场和屏蔽字: 保存现场是指保存程序状态寄存器 PSWR 和某些通用寄存器的内容.
  • 开中断: 允许更高级别中断请求获得响应
  • 执行中断服务程序: 目的
  • 关中断: 保证在恢复现场和屏蔽字时不被打扰
  • 恢复现场和屏蔽字: 将现场和屏蔽字恢复到原来的状态
  • 开中断: 中断返回,返回到原程序断点处继续执行.

系统调用

系统调用的处理需要由操作系统内核程序负责完成,要 运行在核心态 .用户程序可以执行 陷入指令 来发起系统调用,请求操作系统提供服务.

由用户态进入核心态,不仅状态需要切换,而且所用的堆栈也需要由 用户堆栈切换为系统堆栈*.

分类:

  • 设备管理: 完成设备请求或者释放,以及设备启动等功能.
  • 文件管理: 完成文件的读,写,创建及删除功能.
  • 进程管理: 完成进程的创建,撤销,阻塞以及唤醒功能
  • 进程通信: 完成进程间的消息传递或者信号传递等功能.
  • 内存管理: 完成内存分配,会少以及获取作业占用内存区大小以及始址等功能.

进程与线程

基本概念:
进程: 在多道程序环境下，允许多个进程并发执行，此时他们将失去封闭性，并具有间断性及不可再现性的特征。为此引入了进程的概念，以便更好地描述和控制程序的并发执行，
实现:
操作系统的并发性和共享性。
进程是程序的运行过程，是系统进行资源分配和调度的一个独立单位。
线程: 早期，在 OS 中能拥有资源和独立运行的基本单位是进程，然而随着计算机技术的发展，进程出现了很多弊端，一是由于进程是资源拥有者，创建、撤消与切换存在较大的时空开销，因此需要引入轻型进程；二是由于对称多处理机（SMP）出现，可以满足多个运行单位，而多个进程并行开销过大。
线程是操作系统能够进行运算调度的最小单位。它被包含在进程之中，是进程中的实际运作单位。一条线程指的是进程中一个单一顺序的控制流，每条线程执行不同的任务。

进程

进程信息 描述
进程ID 操作系统为每个进程都分配有唯一的 ID 号，方便识别各个进程
进程状态 进程在执行期间可能处于开始、就绪、运行、等待、终止这 5 种状态。
程序计数器 记录进程要执行的下一条指令所在的地址。
CPU 寄存器 不同的计算机体系结构，寄存器的类型和数量也会不同，包括累加器、索引寄存器、堆栈指针等等。当进程发生中断时，进程的状态信息会和程序计数器一起保存，以便一段时间后，恢复该进程继续执行。
内存管理信息 包括进程所占用的内存大小，所在的页表和段表等信息。不同的操作系统所使用的内存系统也略有不同。
I/O 状态信息 分配给进程使用的输入、输出设备，以及进程可操作的文件等等。
其他信息 进程可用的 CPU 的数量、占用 CPU 的时间等等。

创建进程过程

  • 为新进程分配一个唯一的进程标识符,并申请一个空白的 PCB(PCB 不能无限申请),若 PCB 申请失败,则进程创建失败.
  • 为进程分配资源,为新进程的程序和数据以及用户栈分配必要的内存空间,如果资源不足并不会撤销申请,而是会处于阻塞状态等到资源可用继续创建.
  • 初始化 PCB,包括初始化标志信息,初始化处理机状态信息,初始化处理机控制信息,设置进程的优先级等
  • 若进程就绪队列能接纳新进程则将新进程插入就绪队列,等待被调度运行.

进程的终止

  • 正常结束,表示进程的任务已完成并准备退出运行.
  • 异常结束,表示进程在运行时,发生了某种异常时间,使程序无法继续运行,如存储区越界,保护错,非法指令,特权指令错,运行超时,算术运算发错,I/O 故障等.
  • 外界干预,指进程应外界的请求而终止运行,如操作员或操作系统干预,父进程请求和父进程终止.

终止流程

  • 根据进程标识符,检索 PCB,从中读出进程状态.
  • 若被终止进程处于执行状态,立即终止进程的执行,将处理及资源分配给其他进程.
  • 若该进程还有子孙进程,则应将其所有子孙进程终止.
  • 将该进程的所有资源返回给父进程或者操作系统.
  • 将该 PCB 从所在队列(链表)删除.

进程切换

  • 保存处理机上下文,包括程序计数器和其他寄存器.
  • 更新 PCB 信息.
  • 把进程 PCB 移入相应队列,如就绪,在某事件阻塞队列等.
  • 选择另外一个进程执行,并更新 PCB.
  • 更新内存管理的数据结构.
  • 恢复处理机上下文.

进程通信机制

管道

  • 半双工通信,数据只能单向流动.
  • 只能在具有亲缘关系的进程间使用,进程亲缘关系通常指父子进程关系.

共享内存

  • 映射一段能被其他进程访问的内存,这段内存由一个进程创建但是许多进程都可以访问.共享内存是最快的IPC方式.

消息队列

  • 由消息的链表,存放到捏合众并由消息队列标识符标识.
  • 克服了信号传递信息少,管道只能承载无格式字节流与缓冲区大小受限等缺点.

用户线程以及内核线程的区别

用户级线程是指不需要内核支持而在用户程序中实现的线程.内核切换是由用户态程序自己控制内核的切换.

优点:

  • 线程调度不需要内核直接参与
  • 可以在不支持线程的操作系统中实现.
  • 同一进程中只能同时有一个线程在运行,如果有一个线程使用了系统调用而阻塞,那么整个线程都会阻塞.可以节省系统资源.

缺点:

  • 一个用户级线程阻塞将会引起整个进程阻塞.
  • 用户级线程不能利用系统多重处理.仅有一个用户级线程可以执行.

内核级线程切换由内核控制,当线程进行切换时,由用户态转化为内核态.切换完毕要从内核态返回用户态.可以很好的运用多核CPU.

优点:

  • 更好支持多线程,提高效率
  • 切换速度快.

缺点:

  • 占用系统资源多.

物理设备

基本概念

  • 冲突域: 在以太网中,如果某个CSMA/CD网络上的两台计算机在同时通信时会发生冲突,那么则称这个CSMA/CD网络为一个冲突域.
  • 广播域: 网络中某一设备同时向网络中所有其它设备发送数据,这个数据所能光波导的范围即为广播域(Broadcast Domain)

集线器

物理层 设备,对接收到的 MAC 帧不加处理直接进行广播.

网桥

数据链路层 设备,会对接收到的帧进行过滤,只有目的 MAC 地址匹配的数据才会发送到出口.

交换机

数据链路层 设备,可以看做多接口的网桥.交换机内部存在一张 MAC 地址表.该表记录了网络中所有 MAC 地址与该交换机端口的对应关系,
当某一数据帧需要转发时,交换机根据目的 MAC 地址查找 MAC 地址表,从而得到端口

地址表构造过程

假设目前有 E1-5,五台计算机连接交换机.同时交换机的 MAC 地址表为空.
假设目前 E1 想要与 E5 进行通信,当 E1 将帧发送到交换机时交换机发现地址表不存在目的 MAC 地址,
则会将帧进行广播,此时 E5 获得帧并响应,地址表此时会记录下 E5 发送帧的端口(进行绑定),同样的,如果 E1 发送的源地址 MAC 地址表不存在也会记录上去.

交换主要功能

  • 学习: 通过自学习形成 MAC 帧与端口的绑定
  • 转发/过滤: 如果某帧的目的地址存在于 MAC 地址表中,则只会将帧转发到该 MAC 目的地址绑定的端口.
  • 消除回路: 当交换机包括冗余回路时,以太网交换机通过生成树协议避免回路的产生,同时允许存在后备路径.

特性

  1. 交换机的每一个端口所连接的网段都是一个独立的冲突域。
  2. 交换机所连接的设备仍然在同一个广播域内，也就是说，交换机不隔绝广播（惟一的例外是在配有 VLAN 的环境中）。
  3. 交换机依据帧头的信息进行转发，因此说交换机是工作在数据链路层的网络设备（此处所述交换机仅指传统的二层交换设备）。

路由器

工作在 网络层 ,拥有自己独立的MAC地址 路由器的每个端口都有自己的MAC地址 ,可以帮源主机做数据转发.

存储转发流程

  1. 当数据包到达路由器，根据网络物理接口的类型，路由器调用相应的链路层功能模块，以解释处理此数据包的链路层协议报头。这一步处理比较简单，主要是对数据的完整性进行验证，如CRC校验、帧长度检查等。
  2. 在链路层完成对数据帧的完整性验证后，路由器开始处理此数据帧的IP层。这一过程是路由器功能的核心。根据数据帧中IP包头的目的IP地址，路由器在路由表中查找下一跳的IP地址；同时，IP数据包头的TTL（Time To
    Live）域开始减数，并重新计算校验和（Checksum）。
  3. 根据路由表中所查到的下一跳IP地址，将IP数据包送往相应的输出链路层，被封装上相应的链路层报头，最后经输出网络物理接口发送出去。

网络层分组转发流程

  • 从数据报首部提取目的主机的IP地址D,获得目的网络地址N.
  • 若网络N与此路由器直接相连,则吧数据报直接交付给目的主机D,即 路由器的直接交付.
  • 如果路由表中有目的地址为D的特定主机路由(对特定目的主机指明一个特定的路由,通常是为了控制或者测试网络),则把数据报传送给路由表中所指明的下一跳路由器.
  • 如果路由表中有到达N网络的路由,则把数据报传送给路由表中所指明的下一跳路由器,否则执行下一步.
  • 若路由表中有默认路由,贼将数据报传送给默认路由所指明的默认路由器,否则报告分组出错.

三种网络模型

计算机网络 协议,接口,服务的概念

  • 协议: 规则的集合.在网络中要做到有条不紊的交换数据就必须遵循一些事先约好的规则.这些规则明确规定了交换的数据格式以及有关的同步问题.这些为进行网络中的数据交换而建立的规则称为网络协议.他负责控制两个(或者多个)对等实体进行通信的规则的集合.是水平的.不对等实体之间没有协议.
    • 协议主要由语法,语义,和同步三部分组成.
    • 语法 规定了数据传输的格式.
    • 语义 规定了所要完成的功能,即需要发出何种控制信息.完成何种动作及做出何种应答.
    • 同步 规定了执行各种操作的条件,时序关系等.
  • 接口: 同一节点内相邻两层间交换信息的连接点.是一个系统内部的规定.每层只能为紧邻的层次之间定义接口.不能跨层定义接口.在典型接口上,同一结点相邻两层的实体通过服务访问点(Service Access Point SAP)进行交互.服务是通过SAP提供给上层使用的.第n层的SAP就是第n+1层可以访问第n层服务的地方.每个SAP都有一个能够标识它的地址.SAP是一个抽象的概念.他实际上是一个逻辑接口,但和通常所说两个设备之间的硬件接口是很不一样的.
  • 服务 指下一层为紧邻的上层提供的功能调用.对等实体在协议控制下,使得本层能够为上一层提供服务.但要实现本层协议还需要使用下一层所提供的服务.
    • 请求 由服务用户发往服务提供者,请求完成某项工作.
    • 指示 由服务提供者发往服务用户,指示用户做某件事情.
    • 响应: 由服务用户发往服务提供者,作为对指示的响应.
    • 证实 由服务提供者发往服务用户,作为对请求的证实.

OSI 七层模型

  • 应用层: 为特定类型的网络应用提供访问 OSI 参考模型环境的手段.

  • 表示层: 处理在两个通信系统中交换信息的表达方式.

  • 采用抽象标准方法定义数据结构.

    • 采用标准的编码方式,数据压缩,加密和解密也是表示层的功能.

  • 会话层: 允许不同主机上的进程进行通话.

    • 负责 管理主机间的会话进程 ,包括建立,管理以及终止进程间的会话.

    • 会话层可以使用校验点在通信失效时恢复通信.

  • 运输层: 负责主机中两个进程之间的通信,

    • 为端到端链接提供可靠的传输服务
    • 为端到端链接提供 流量控制 ,服务质量 ,差错控制 ,数据传输管理 等功能.

  • 网络层:

    • 把网络层协议数据单元从源端传输到目的端
    • 为分组交换网上不同主机提供通信服务.
    • 对分组进行 路由选择 ,并实现 拥塞控制.

  • 数据链路层:

    • 传输单位是帧,,任务是将网络层传来的 IP 数据包封装成帧.
    • 成帧,差错控制,流量控制,传输管理

  • 物理层:

    • 在物理媒体上为数据段设备透明传输原始 byte 流

TCP/IP 模型

  • 应用层:
    • 包含所有高层协议,DNS,Telnet,FTP 等
  • 传输层:
    • TCP(控制传输协议 Transmission Control Protocol):
      • 面向连接,数据传输单位是报文段,能够提供可靠传输
    • UDP(User Datagram Protocol 用户数据报协议):
      • 无连接,提供尽最大努力交付
  • 网际层:
    • 定义了标准的分组格式和协议,即 IP
    • 不保证分组有序到达,各分组有序交付由高层决定.
  • 网络接口层:
    • TCP/IP 未真正描述该部分,只是指出主机必须使用某种协议与网络连接,以便在其上传递 IP 分组.

C/S模型

C/S模型(Client/Server)模型
改模型中有一个总是打开的主机称为服务器,它服务于许多来自其他称为客户机的主机请求.

以下为工作流程:

  • 服务器处于接受请求的状态.
  • 客户机发出服务请求,并等待接受结果.
  • 服务器接受请求后,分析请求,进行必要的处理.得到结果并发送给客户机.

P2P模型

整个网络中的传输内容不再被保存到中心服务器上,每个节点都同时具有下载,上传的功能.

以下为特点:

  • 去中心化: p2p是分布式系统,网络招你的资源和服务分散在所有节点中,信息的传输和服务的实现都直接在节点之间进行,可以无需中间环节 和服务器介入.
  • 可扩展性 用户可以随时加入网络,系统资源和服务能力随之扩展.
  • 健壮性 因为服务是分散在各个节点之间的,部分节点或者网络遭到破坏对其他部分的影响很小
  • 高性价比 计算机的任务或者资料分布到所有节点,达到了高性能计算和海量存储的目的.
  • 隐私保护 用户的信息被分散到各个节点之间,降低了用户信息的泄露.
  • 负载均衡 每个节点既是服务器又是客户端,减少了传统C/S模型中对服务器计算能力,存储的要求.将这种要求分布在各个节点之上.

DHCP

应用层协议,使用UDP作为传输层协议

DHCP流程:

  • DHCP客户广播”DHCP发现”消息 试图找到网络中的DHCP服务器.以便从DHCP服务器获得一个IP地址.
  • DHCP服务器收到”DHCP发现”消息, 向网络中广播”DHCP提供”消息,其中包括提供DHCP客户机的IP地址和相关配置信息.
  • DHCP客户机收到”DHCP提供”信息, 如果接受DHCP服务器提供的相关参数,则通过广播”DHCP请求”消息向DHCP服务器请求提供IP地址.
  • DHCP服务器广播”DHCP确认”信息 将IP地址分配给DHCP客户机.
DNS

DNS是因特网使用的命名系统,用来把便于人们记忆的具有特定含义的主机名转化为便于及其处理的IP地址. 运行在UDP之上,使用53号端口.

域名解析流程
递归查询:

  • 本地域名服务器只需向根域名服务器查询一次,后面几次查询都是递归地在其他几个域名服务器之间进行
  • 缺点 服务器会产生过大负载.

    迭代查询
  • 当根域名服务器收到来自本地域名服务器的迭代查询请求时,要么给出所要查询的IP地址,要么告诉本地域名服务器下一步该往哪个服务器发送请求.
FTP

文本传输协议(File Transfer Protocol) 提供交互式访问,允许客户指明文件的类型与格式.并允许文件具有存取权限.
适合在异构网络任意计算机中传输文件.
特点:

  • 提供不用种类主机系统(软硬体系都可以不同)的文件传输服务.
  • 以用户权限管理的方式提供用户对远程FTP服务器上的文件管理能力
  • 以匿名FTP方式提供公用文件共享能力.

工作时使用两个并行TCP连接,一个用于数据传输(端口号20),一个用于命令传输(端口号21).
控制连接:

  • 服务器监听21号端口,等待客户连接.建立于这个端口上的连接称为控制连接.控制连接用来传输控制信息(如连接请求,传送请请求等), 并且控制信号都以7位ASCII码方式传输.

  • FTP客户发出传送请求,通过控制连接发送给服务器端的控制进程.但控制连接并不用来传送文件,在传输文件时可以使用控制连接(如客户在传输过程中发送了终止命令),因此控制连接在整个会话期间一直保持打开状态 .

数据连接:

服务器端控制进程在接收到FTP客户发来的文件传输请求时,创建”数据传送进程“和”数据连接进程“.
数据连接用来连接客户端和服务器端的数据传送进程,数据传送进程实际完成文件的传送,在传送完毕后关闭”数据传送连接”并结束运行.

数据连接存在两种方式,一种主动模式PORT,一种被动模式PASV.

主动模式工作原理:

  • 客户端连接到服务器的21号端口,登陆成功后读取数据时,客户端随机开放一个端口并发送命令告知服务器,服务器接收到PORT命令和端口号后,通过20端口和客户端开发的端口连接,发送数据.

PASV与PORT的区别:

  • 客户端读取数据时,发送PASV命令告诉服务器,服务器在本地随机开放一个端口,并告知客户端,客户端再连接到服务器开放端口进行数据传输.
电子邮件协议

发送邮件流程:

  • 发信人调用用户代理撰写和编辑要发送的邮件.用户代理用SMTP把邮件传输给发送端邮件服务器.
  • 发送端邮件服务器将邮件放入邮件缓存队列中等待发送.
  • 运行在发送端邮件服务器的SMTP进程检查邮件缓存队列,如果存在邮件就向运行在接收端邮件服务器的SMTP客户进程发起建立TCP连接.
  • TCP连接建立后,SMTP客户端进程开始向远程SMTP服务器进程发送邮件,当所有待发送邮件发完后,SMTP就关闭所建立的TCP连接.
  • 运行在接收端邮件服务器中的SMTP服务器进程接收到邮件后,将邮件放入收信人的用户邮箱,等待收信人在方便时进行读取.
  • 收信人打算收信时,调用用户代理,使用POP3(or IMAP)协议将自己的邮件从接收端邮件服务器的用户邮箱中取回.
HTTP协议

面向事务的应用层传输协议,规定了在浏览器和服务器之间的请求和响应的格式与规则,是万维网上能够可靠地交换文件的重要基础.

流程:

  • 浏览器要访问www服务器的时候,首先要完成对WWW服务器的域名解析,一旦获得服务器的IP地址,就可以通过TCP向服务器发送建立请求.
  • 服务器有一个监听进程,不断监听TCP80端口,当监听到连接请求时就会建立TCP连接,构建响应并返回给客户端.

HTTP特点:

  • 使用TCP作为传输层协议,保证数据可靠性.
  • 无状态 同一个客户第二次访问同一个服务器上页面时,获得的响应与第一次相同.
  • 既可以是持久化连接,也可以是非持久化连接(HTTP/1.1)

软件工程

什么是软件危机:

表现 对软件开发成本和时间估计不准,用户对已完成软件不满意,软件质量不可靠.软件不可维护,软件缺少文档
软件成本过高,软件跟不上硬件发展速度.

原因: 与软件本身有关,缺乏可见性,质量难以评价,规模庞大难以维护.与软件开发维护的方法不当有关.
轻视需求分析和维护,对用户的要求没有完整准确的认识就编写程序.

什么是软件工程:

采用工程的概念,原理,技术和方法开发和维护软件,把正确的管理技术和软件开发技术结合起来,经济地开发出高质量软件并有效地维护.
即把系统化的,规范的,可度量的途径应用于软件开发,运行和维护.

软件工程七条基本准则:

软件生命周期(SDLC,软件生存周期)是指软件自生产到报废的周期.为了使规模大,结构复杂,管理复杂的软件开发变得容易控制和管理.人们把软件生命周期划分为若干个阶段.
使每个阶段都由明确的任务.整理出软件生命周期模型.在 1980 年人类整理了第一个软件生命周期,即瀑布模型.

软件质量的理解:

软件质量是指”软件与明确的和隐含的定义的需求相一致的程度,具体来说,软件质量是软件符合明确叙述的功能和性能需求,文档中明确描述的开发标准,以及所有专业开发的软件都应具有的隐含特征的程度.

  1. 软件需求是度量软件质量的基础.与需求不一致就是质量不高.
  2. 指定的标准定义了一组知道软件开发的准则.
  3. 通常,有一组没有显式描述的隐含需求(如期望软件是容易维护的),如果软件满足明确描述的需求,但却不满足隐含的需求,那么软件的质量仍是值得怀疑的.

生命周期模型

瀑布模型

把一个开发过程分成收集需求,分析,设计,编码,测试,维护六部分.
只有完成前面一步才能开始后面一步,上一步的输出的文档就是这一步的输入文档,每一步完成都要交出合格的文档,每一步都会有反馈,如果反馈有错误就退回前一步.
瀑布模型分为 可行性分析与开发项计划,需求分析,设计(概要设计和详细设计),编码,测试,维护等阶段.

  • 问题的定义以及规划: 软件开发方和需求放过共同讨论,主要确定软件的开发目标以及可行性.
  • 需求分析: 在确定软件开发可行的情况下,对软件需要实现的各个功能进行详细分析.
  • 软件设计: 主要根据需求分析的结果,对整个软件系统进行设计.
  • 程序编码: 将软件设计的结果转换成计算机可运行的程序代码.
  • 软件测试: 进行严格软件测试,以便发现整个设计过程中存在的问题并加以纠正.
  • 运行维护: 由于多方面原因,如果软件不能继续适应用户的要求.要延续软件的使用寿命,就必须要对软件进行维护.软件的维护包括纠错性维护和改进型维护两个方面.
    缺点:

实际项目开发很难严格按照该模型进行:由于用户只能通过文档来了解产品,客户往往很难清楚地给出所有需求.
瀑布模型不适应用户需求的变化,软件的实际情况必须到项目开发的后期客户才能看到.

V 模型

V 模型大致可以分为以下几个步骤:需求分析,概要设计,详细设计,软件编码,单元测试,集成测试,系统测试,验收测试.

  • 需求分析: 明确产品的各项属性以及功能.关键在于设计师与客户沟通,获取准确详细的设计,功能,实现方式.并转换为需求规格设计书.
  • 概要设计: 主要是架构实现,指 搭建架构,表述各模块功能,模块接口链接和数据传递的实现等事务.
  • 详细设计: 对概要设计中各模块进行深入分析,对各模块组合进行分析等.要求达到伪代码级别,把程序的具体实现的功能,现象表述出来.该阶段需要包括数据库设计说明.
  • 软件编码: 按照详细设计好的模块功能表,编程人员编写出实际的代码.
  • 单元测试: 按照设定好的最小测试单元进行单元测试,主要是测试程序代码,确保各模块被正确编译,单元的具体划分按不同的单位与不同的软件.
  • 集成测试: 经过单元测试后,将各个单元组成完整体系.主要测试各模块间组合后的功能实现情况,以及模块接口链接的成功与否.数据传递的正确性等.
  • 系统测试: 按照软件规格说明书要求,测试软件其性能功能等是否和用户需求相符合.在系统中运行是否有漏洞等.
  • 验收测试: 主要在用户拿到软件时根据需求说明书进行响应测试.

增量模型

将软件分为多个构件,每个构件分别当做一个软件来 Fenix,设计,编码,测试.开发人员一次一个构件的提交给用户,最后组装到一起.

优点: 一开始不需要投入太多人力,如果核心产品很受欢迎,则可增加人力实现下一个增量.可先发布部分功能给客户,让客户逐渐适应.

缺点: 并行开发构件可能遇到不能集成的风险,软件必须具备开放式的体系结构.

螺旋模型

在快速圆形模型的每一个原型前引入一个非常严格的风险分析,每个原型经过制定计划,风险分析.实施工程,客户评估四步走,每个原型都标识一个或多个主要风险.
知道所有主要因素因素都被确定.螺旋模型由风险驱动,强调可选方案和约束条件从而支持软件的重用.有助于将软件质量作为特殊目标融入产品开发中.

  • 优点: 设计上灵活可以在项目的各个阶段进行更改,以小的分段来构件大型系统,是成本计算变得简单容易,客户始终参与每个阶段的开发,保证了项目不偏离正确方向爱过你以及项目的可控性.
  • 缺点: 采用螺旋模型需要具有想当丰富的风险评估经验和专门知识,在风险较大的项目开发中如果未能够及时标识风险,势必造成重大损失.

喷泉模型

以用户需求为动力,对对象为驱动的模型,主要用于描述面向对象的软件开发过程.改模型认为软件开发过程自下而上周期的各阶段是相互迭代和无间隙的特性.喷泉模型不像瀑布模型,需要分析活动结束后才开始设计活动.设计活动结束后才开始编码活动.改模型的各个阶段没有明显的界限.开发人员可以同步进行开发.

  • 优点: 可以提高软件项目开发效率,节省开发时间,是英语面向对象的软件开发过程.
  • 缺点: 喷泉模型在各个开发阶段重叠,所以在开发过程中需要大量的开发人员.

rational 统一过程

是一个软件开发过程框架,值团队可以根据具体项目组或软件开发企业的不同需求能够定义,配置,定制和实施一致的软件开发过程.
包括三核心: 用于成功开发软件的一组基本观念和原则,一套关于可重用方法内容和过程构件的框架,基础的方法和过程定义语言.

RUP 最佳实践

描述了一个指导开发团队达成目标的迭代式的软件开发过程.包括:
迭代式开发(每次迭代过程中需求改变,每个迭代过程以完成可执行版本结束,让用户不断介入和提出意见并改进),
管理需求(用脚本来确定需求),使用基于组件的架构(使用功能清晰的模块),可视化建模(可视化图形更容易令人理解),帮助人们提高管理软件复杂性的能力,验证软件质量(软件质量评估贯穿整个开发过程),控制软件变更(确保每个更改都是可接受和可追踪的).

敏捷开发

以用户的需求进化为核心,采用迭代,循序渐进的方法进行软件开发,在敏捷开发中,软件项目在构建初期被切分为多个子项目各个子项目成果都进过测试,具备可视,可集成和可运行使用特点

极限编程

把好的开发时间运用到极致,广泛运用模糊且经常改变的场合,过程.首先,项目组针对客户代表提出的要求进行讨论,对体系结构进行试探,然后项目根据客户设定的优先级制定交付计划.
然后开始多个迭代过程,迭代期中新用户故事不在本次迭代内解决,开发出的新版本软件通过验收测试之后交付用户使用,极限编程可以迅速快捷地响应变化.同时保证开发速度.

迭代过程: 项目组根据项目速率,选择需要完成的用户需求或者应该消除的差错,制定出新的本次迭代计划,每天举行一次站立会议解决遇到的问题,调整迭代计划.开发出的新功能必须通过单元测试,立即继承,得到新的可运行版本由客户代表进行验收测试,客户代表提出问题,项目组根据问题提出新的用户要求,算出新的项目速率.再讨论.

结构化设计

软件设计的目标:

  • 正确性: 满足应用程序需求
  • 健壮性: 软件对于规范要求意外的输入情况的处理能力.也就是说,在异常情况下软件能够正常运行的能力.
  • 灵活性: 允许代码修改平稳地发生,而不会被波及到很多其他的模块.
  • 可重用性: 重复使用
  • 高效性: 1.执行效率,2.存储效率

软件设计的原理

  • 模块化 (把程序划分为独立命名独立访问的模块,每个模块完成一个子功能)
  • 抽象 (高层次抽象一开始以概括的方式叙述问题的解法,然后低层次抽象把面向问题的术语和面向显示的术语结合起来叙述问题的解法,最后再最底层用直接实现的方式叙述问题的解法)
  • 逐步求精: 即将系统功能按层次进行分解,每一层不断将功能喜欢上,到最后一层都是功能单一,简单易实现的模块.

模块独立

独立的模块比较容易开发，测试和维护，耦合(一个软件结构内不同模块之间的互联程度，耦合的程度取决于模块间接口和复系程度），内聚（一个模块内各个元素彼此结合的紧密程度），我们追求的是高内聚和低耦合

高内聚低耦合

定义:
内聚是从功能角度来度量模块内的联系，一个好的内聚模块应当恰好做一件事。它描述的是模块内的功能联系：
耦合是软件结构中各模块之间相互连接的一种度量，耦合强弱取决于模块间接口的复杂程度、进入或访问一个模块的点以及通过接口的数据。

概念:
耦合性： 也称块间联系。指软件系统结构中各模块间相互联系紧密程度的一种度量。模块之问联系越紧密，其耦合性就越强，模块的独立性则越差。模块间耦合高低取决于模块间
接口的复杂性、调用的方式及传递的信息内聚性：又称块内联系。指模块的功能强度的度量，即一个模块内部各个元素彼此结合的紧密程度的度量。若一个模块内各元素 (语名之间、程序段之问）联系的越紧密，则它的
内聚性就越高。
所谓高内聚是指一个软件模块是由相关性很强的代码组成，只负责一项任务，也就是常说的单一责任原则.

耦合： 一个软件结构内不同模块之间互连程度的度量。对于低耦合，粗浅的理解是：一个完整的系统，模块与模块之间，尽可能的使其独立存在。也就是说，让每个模块，尽可能的独立完成某个特定的子功能。模块与模块之间的接口，尽量的少而简单。如果某两个模块间的关系比较复杂的话，最好首先考虑进一步的模块划分。这样有利于修改和组合。

分类:
耦合可以分为以下几种，它们之间的耦合度由高到低排列如下：

  • 内容耦合：
    一个模块直接访问另一模块的内容，则称这两个模块为内容耦合。 内容糯合可能在汇编语言中出现。大多数高级语言都己设计成不允许出现内容耦合。 这种耦合的耦合性最强，模块独立性最弱。
  • 公共耦合： 一组模块都访问同一个全局数据结构，则称之为公共耦合。公共数据环 境可以是全局数据结构、共享的通信区、 内存的公共覆盖区等。如果模块只是向公共数据
    环境输入数据，或是只从公共数据环境取出数据，这属于比较松散的公共耦合：如果模块 既向公共数据环境输入数据又从公共数据环境取出数据，这属于较紧密的公共耦合。
  • 外部耦合： 一组模块都访问同一全局简单变量，而且不通过参数表传递该全局变量 的信息，则称之为外部耦合。
  • 控制耦合： 模块之问传递的不是数据信息，而是控制信息例如标志、开关量等， 个模共控制了另一个模块的功能。
  • 标记耦合： 调用模块和被调用模块之间传递数据结构而不是简单数据，同时也称作 特征耦合。表就和的模块问传递的不是简单变量，而是像高级语言中的数据名、记录名和 文件名等数据结果，这些名宇即为标记，其实传递的是地址。
  • 数据耦合： 调用模块和被调用模块之间只传递简单的数据项参数。相当于高级语言 中的值传递。
  • 非直接耦合： 两个模块之间没有直接关系，它们之间的联系完全是通过主模块的控 制和调用来实现的。耦合度最弱，模块独立性最强。

总结： 有好人是影响软件复杂程度和设计质量的一个正要因素，为提商棋块的独立性， 应建立模块间尽可能松散的系统，在设计上我们应采用以下原则：若模块间必须存在耦合，应尽量减少数据耦合，少用控制耦合
，慎用或有控制地使用公共耦合，并限制公共藕 合的范围，尽量避免内容耦合。 内聚有如下的种类，它们之间的内骤度由弱到强排列如下：

  • 偶然內聚： —个模块内的各处理元素之问没有任何联系，只是偶然地被湊到一起。 这种模块也称为巧合内聚，内聚程度最低。
  • 选择内聚： 这种模块把几种相关的功能组合在一起， 每次被调用时，由传送给模块 参数来确定该模块应完成哪一种功能
  • 时间内聚： 把需要同时执行的动作组合在一起形成的模块称为时间内聚模块。
  • 过程内聚： 构件或者操作的组合方式是，允许在调用前面的构件或操作之后，马上 调用后面的构件或換作，即使两者之问没有数据进行传递。简单的说就是如果一个模块內 的处理元素是相关的，而且必须以特定次序执行则称为过程内聚。
  • 通信内聚： 指模块内所有处理元素都在同一个数据结构上操作或所有处理功能都通 过公用数据而发生 关联（有时称之为信息内聚）。即指模块内各个组成部分都使用相同的 数据数据或产生相同的数据结构。
  • 顺序内聚：
    一个模块中各个处理元素和同一个功能密切相关，而且这些处理必须顺 序执行，通常前一个处理元素的输出是后一个处理元素的输入。 例如某模块完成工业产值求值的功能，前一个功能元素求总产值，后一个功能元素求 平均产值，显然该模块内两部分紧密关联
    顺序内聚的内聚度比较高，但缺点是不如功能内聚易于维护。
  • 功能内聚： 模块内所有元素的各个组成部分 全部都为完成同一个功能而存在，共同 完成一个单一的功能，模块己不可再分。即模块仅包括为完成某个功能所必须的所有成 分，这些成分紧密联系、缺一不可。
    功能内聚是最强的内聚，其优点是它的功能明确。判断一个模块是否功能内聚，一般从模块名称就能看出。 如果模块名称只有一个动词和一个特定的目标（单数名词），一般来说就是功能内聚，如：“计算水费”、“计算产值”
    等模块。功能内聚一般出现在软件结构图的较低层次上。 功能内聚模块的一个 重要特点 是：他是一个”暗盒”，对于该模块的调用者来说，只需 要知道这个模块能做什么，而不需要知道这个模块是如何做的。 总结：在模块划分时，要避循”
    一个模块，一个功能”的原则，尽可能使模块达到功能内聚。

    高内聚，低耦合的系统有什么好处呢？事实上，短期来看，并没有很明显的好处，甚至短期内会影响系统的开发进度，因为高内聚，低男合的系统对开发设计人员提出了更高
    的要求。高内聚，低男合的好处体现在系统持续发展的过程中，高内聚，低男合的系统具有更好的重用性，维护性，扩展性，可以更高效的完成系统的维护开发，持续的支持业务
    的发展，而不会成为业务发展的障碍。

编译原理

什么是编译器?
一个编译器就是一个程序,可以阅读一种语言编写的源程序并将其转换为等价地另外一种语言编写的源程序.
编译器的重要任务就是在翻译过程中发现源程序中出现的问题.如果目标程序是一个可执行的机器语言程序,那么它就可以被用户调用,处理出入并产生输出.

什么是解释器?

解释器是另外一种常见的语言处理器.并不是通过翻译的方式生成目标程序,从用户角度看,解释器直接利用用户提供的输入执行源程序中指定的操作.

编译器编译流程

  • **词法分析:**任务是扫描源程序,根据语言词法规则分解和识别出每个单词.并把单词翻译为对应的机内表示.

  • **语法分析:**在词法分析的基础上进行的,任务是根据语言的语法规划吧单词符号流分解为格内语法单位.比如表达式,语句等.通过语法分析确定整个输入符号流是否构成一个语法正确的程序.

  • 语义分析: 对源程序进行语义检查.目的是保证标识符和常数的正确使用.把必要的信息手机保存到符号表或者中间代码程序中,并进行相应的处理.

  • 中间代码生成: 必不可少的阶段.任务是在语法分析和语义分析基础上把语法成分的语义对其继续翻译,翻译的结果是某种中间代码形式.这种中间代码结果简单,接近计算机的指令形式,能够很容易被翻译成计算机指令,常用的中间代码有三元式,四元式和逆波兰式.

  • 代码优化 对程序代码进行等价(指不改变程序的运行结果)变换,程序代码可以使中间代码(如四元式代码).也可以是目标代码.等价的含义是是的变换后的代码运行结果与变换前的代码运行结果相同.优化的含义是指最终生成的目标代码短,时空效率优化.

  • 目标代码生成: 将中间代码或者优化之后的中间代码转换成等价的目标代码.即机器指令或者汇编指令.

堆和栈的区别

  • 栈:
    • 在window下栈是向低地址扩展的数据结构,是一块连续的内存区域,栈大小固定,如果申请的空间超过栈剩余空间时,将提示overflow
    • 栈由系统自动分配.速度快,但是程序员无法控制.
    • 存储内容不同 在函数调用期间,栈中存放的是函数中各个参数(局部变量).栈底是函数调用后的下一条指令.
  • 堆:
    • 高地址扩展的数据结构,是不连续的内存区域.这是由于系统使用链表来存储空闲内存地址的，自然是不连续的，而链表的遍历方向是由底地址向高地址。堆的大小受限于计算机系统中有效的虚拟内存。
    • 堆是由malloc分配的内存,一般速度比较慢,而且容易产生碎片,但是使用方便.需要用户手动释放空间,使用不当会导致内存泄露.
    • 一般在堆头部用一个字节存放堆大小,堆中具体内容由程序员决定.

虚函数

C++多态是通过虚函数实现的,虚函数允许自雷重新定义成员函数,而子类重新定义父类的做法称为覆盖 (override)
简而言之是通过父类型别的指针指向其子类的实例.然后通过父类指针调用实际子类成员函数.该技术可以让父类指针有多重形态.

面向对象的概念

基本特性

  • 抽象:
  • 封装:
    • 把对象属性和行为看成一个密不可分的整体,将这两者封装在一个不可分割的独立单元中-
    • 把不需要让外界知道的信息隐藏起来,有些对象的属性以及行为允许外界用户知道或者使用,但是不允许更改.而另外一些属性或者行为不允许外界知晓,或者只允许使用功能.-
    • 优点:
      • 减少耦合,符合程序设计追求的”高内聚,低耦合”
      • 类内部结构可以自由更改.
      • 可以对成员变量进行更精确的控制.
      • 隐藏信息实现细节.
  • 继承:
    • 允许创建分层次的类
    • 所谓继承就是子类继承父类的特征和行为.使得子类对象(实例)具有父类的实例与方法,或者子类从父类继承方法,使得子类具有父类相同的行为和方法.
    • 优点:
      • 提高了类代码的复用性
      • 维护性

    • 在JAVA中表现为:
      • 存在继承关系.
      • 子类重写了父类方法.
      • 父类数据类型引用指向了子类对象.
    • JAVA中向上转型运行时动态绑定类型.
      • 虚拟机提取对象市级类型的方法表.
      • 虚拟机搜索方法签名.

基本概念

  • 时间复杂度: 一般情况下,算法中基本操作重复执行的次数是问题规模n的某个函数$f(n)$,算法的时间度量记为$T(n)=O(f(n))$.他表示随问题规模n的增加,算法执行时间的增长率和f(n)的增长率相同,乘坐算法的时间复杂度.被称为问题的基本操作的原操作应该是其重复执行次数和算法的执行时间成正比的原操作.多数情况下爱它是算法最深层循环内的语句的原操作.
  • 空间复杂度: 类似于算法的时间复杂度.$S(n)=O(f(n))$.一个程序除了需要需要存储空间来寄存本身所用指令常数等,还需要一些对数据进行操作的工作单元和存储一些为实现计算所需信息的辅助空间.
